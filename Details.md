# SpendSense AI — Deployment & Architecture Details

This document outlines the exact architecture mapping required for deploying the SpendSense AI full-stack application on a free hosting platform or production environment.

## Exact Architecture for Deployment Mapping

Our architecture runs as a unified full-stack application where the Express backend (`server.ts`) serves both the REST API endpoints and routes static frontend assets compiled by Vite.

- **Frontend Build Output:** The application is compiled into static assets placed in the `dist` folder.
- **Backend Entry Point:** The production Node.js server entry point will be `node dist/server.cjs` or equivalent executing `server.ts` routes. All traffic should route through **port 3000**.
- **External Dependencies:** You must provision a remote **MongoDB instance** to handle data persistence securely.

### Custom NGINX / Express 400 Bad Request Fallback

To bypass proxy errors from reverse-proxies (like NGINX in production) and handle unpredictable client payload formats securely, we engineered a custom Express fallback for `400 Bad Request`.

Our JSON body parsers are wrapped in custom error-handling middleware. If a malformed payload or unsupported body occurs, instead of throwing an unhandled exception that causes structural proxy gateway dropouts (e.g. `502 Bad Gateway` from NGINX), the Express router catches the malformed parsing and elegantly falls back to returning a structured `400 Bad Request` JSON response. This ensures proxy layers correctly interpret the response as client-sided error handling rather than backend failure.

### Database-Backed Rolling Window Pruning Logic

To keep database storage costs minimal while maintaining a premium user experience, we employ database-backed rolling window auto-pruning.

1. **40-Message Chat Pruning:** 
   After saving a new model message response generated by Gemini, a secondary operation runs against the `ChatMessage` collection.
   - We query the total `messageCount` for the current user.
   - If greater than 40, we use MongoDB aggregation `.sort({ createdAt: -1 }).limit(40)` to isolate the IDs of the 40 most recent valid messages.
   - Using `$nin`, we invoke `deleteMany` for any chat document IDs falling outside this safe window.

2. **7-Notification Pruning Rule:**
   Upon creating a new system/alert notification for a user, a similar rolling-window cleanup removes the oldest notifications.
   - We maintain a strict count of max 7 notifications per user.
   - A `.find({ userId }).sort({ createdAt: -1 }).skip(7)` query identifies the exact documents exceeding the limit immediately upon appending a new one.
   - The overflow documents are cleared out via `deleteMany`, ensuring the UI inbox remains highly relevant and lightweight.

## Required Environment Variable Definitions

When deploying to a production host, ensure the following environment variables are securely injected into your deployment configuration:

- `NODE_ENV`: Set to `production`.
- `PORT`: Explicitly set to `3000`.
- `MONGO_URI`: The MongoDB connection string pointing to your primary cluster.
- `GEMINI_API_KEY`: The production API key for `@google/genai` model requests.
- `CRON_SECRET`: A secure bearer token necessary to authorize internal scheduled CRON routes.

## Core Route Endpoints

The deployment platform must map HTTP methods properly to the following API endpoints exposed within the application:

### Webhooks & CRON
- `POST /api/cron/monthly-digest`: Requires `CRON_SECRET` authorization. Iterates users to dispatch summary insights.
- `POST /api/cron/market-brief`: Requires `CRON_SECRET`. Analyzes the market with Gemini and broadcasts to pro users.

### AI Engine & Analytics
- `POST /api/wealth/generate`: The AI Portfolio Architect endpoint for calculating and streaming personalized ETF allocations.
- `GET /api/chat/history`: Fetches up to the 40 most recent chat messages (rolling DB window).
- `POST /api/chat`: Handles user prompts, injects Context, routes through Gemini, and applies auto-prunes.

### Notification API
- `GET /api/notifications`: Retrieves current user's live unread elements.
- `POST /api/notifications/read`: Marks targeted or all notifications as read.

### Protection Logic
- `POST /api/policy/scan`: Upload handler and AI Auditor for evaluating hidden policy clauses.
- `POST /api/policy/enforce`: Claim Enforcer generator for email drafts.
